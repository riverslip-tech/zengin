## 1. システム概要

### 1.1 目的
請求書PDFから振込先情報を抽出し、銀行に提出可能な「全銀フォーマット」テキストファイル（120バイト固定長レコード）を生成する業務ツール。

### 1.2 解決する課題
- 請求書PDFから手作業で振込データを起こす作業を自動化
- 半角カナ・固定長・コード解決などの全銀仕様への準拠を機械的に保証
- 振込元・取引先マスタの重複入力を排除
- 月次振込の履歴を保持し、再生成・再ダウンロードを容易にする

### 1.3 想定ユーザー
- 経理担当者（請求書PDFを受け取り、銀行に振込ファイルを提出する立場）
- 非エンジニア（GUI/Webブラウザでの操作のみで完結する設計）

## 2. 機能要件

### 2.1 必須機能（実装済み）

| ID | 機能 | 概要 |
| --- | --- | --- |
| F-01 | PDF複数アップロード | Webブラウザから1枚以上の請求書PDFを送信。後から追加可。 |
| F-02 | 自動抽出 | Claude API (PDFビジョン) で受取人名・銀行・支店・口座番号・金額を抽出。失敗時は正規表現でフォールバック。 |
| F-03 | 銀行・支店プルダウン | zengin-code パッケージから全1,152銀行・全支店データを取得し、検索可能なドロップダウンで選択。コード・半角カナ名を自動補完。 |
| F-04 | 振込先編集 | 各PDFのプレビュー画像と編集フォームを横並びで表示。サマリー一覧テーブルでの一括編集も可能。 |
| F-05 | 取引先マスタ | MoneyForward「取引先・取引先口座・支払先」エクスポートCSVを取り込み、検索・編集・削除。 |
| F-06 | 自動マッチング | 抽出結果を口座番号→名義人カナ→取引先名の優先順でマスタと照合し、一致したらマスタの値で上書き。 |
| F-07 | プロンプトヒント | マスタの取引先名一覧を Claude のシステムプロンプトに動的追加し、抽出精度を向上。 |
| F-08 | マスタ未登録の登録 | 未マッチの取引先には「マスタに登録」ボタンを表示。ダイアログでワンクリック登録。 |
| F-09 | 振込元複数管理 | 振込元（委託者）情報を複数登録・切り替え可能。デフォルト指定可。 |
| F-10 | 振込バッチ永続化 | 「YYYY年MM月DD日振込分」単位でバッチを自動保存。PDFバイナリ・編集状態・振込元紐付けを SQLite に保存。 |
| F-11 | 過去バッチ復元 | 起動時に直近バッチを自動ロード。過去バッチをプルダウンから開く・リネーム・削除。 |
| F-12 | 全銀ファイル生成 | 120バイト固定長レコードを cp932 で書き出し。同一振込先（銀行+支店+口座番号+種目）は金額を合算。 |
| F-13 | ワンクリック自動ダウンロード | 生成→HTML注入でブラウザのダウンロードを自動開始。 |
| F-14 | パスワードゲート | Streamlit Cloud上でのアクセスを `APP_PASSWORD` Secret で保護。 |
| F-15 | 差分検証ツール | 過去の正規ファイルと生成ファイルを120バイト単位で比較する CLI（`webapp/compare_zengin.py`）。 |

### 2.2 振込種別
全銀フォーマットの種別コードを以下にマッピング:
- 総合振込 (`21`)
- 給与振込 (`11`)
- 賞与振込 (`12`)

## 3. 非機能要件

### 3.1 性能
- PDF 1枚あたり Claude API 抽出 3〜8秒（モデル: `claude-sonnet-4-6`）
- 20件程度の請求書を3〜5分以内で全銀ファイル化

### 3.2 信頼性
- 全レコード必ず120バイトを `assert` で保証
- cp932エンコーディング不可文字は事前にサニタイズして除去
- 抽出失敗（API応答異常）時も空のInvoiceで枠だけ作り、手動入力で補完可能

### 3.3 保守性
- コアロジック（`src/zengin_converter/`）はWeb UIに依存しないPure Python
- UIロジック（`webapp/`）から `import zengin_converter` でコア機能を再利用
- 3経路（CLI / tkinter GUI / Streamlit Web）から同じビルダーを呼ぶ

### 3.4 セキュリティ
- APIキー、振込元口座、取引先マスタは `.gitignore` で除外
- Streamlit Cloud では Secrets に保管
- パスワードゲートで未認証アクセスをブロック

## 6. 全銀フォーマット仕様

### 6.1 ファイル構造
4種類のレコードを連結。**1レコード = 120バイト固定長、cp932エンコーディング、区切り文字なし**。

```
[ヘッダ(1)] [データ(2)] × N件 [トレーラ(8)] [エンド(9)]
```

### 6.2 ヘッダレコード（区分=1）

| フィールド | 種別 | 長さ | 内容 |
| --- | --- | --- | --- |
| データ区分 | N | 1 | "1" |
| 種別コード | N | 2 | "21"/"11"/"12" |
| コード区分 | N | 1 | "0" (JIS) |
| 委託者コード | N | 10 | 銀行から付与の10桁 |
| 委託者名 | C | 40 | 半角カナ、左詰スペース埋め |
| 振込日 | N | 4 | MMDD |
| 仕向銀行番号 | N | 4 |  |
| 仕向銀行名 | C | 15 | 半角カナ |
| 仕向支店番号 | N | 3 |  |
| 仕向支店名 | C | 15 | 半角カナ |
| 預金種目 | N | 1 | 1:普通 2:当座 4:貯蓄 9:その他 |
| 口座番号 | N | 7 | 右詰ゼロ埋め |
| ダミー | C | 17 | スペース |

### 6.3 データレコード（区分=2）

| フィールド | 種別 | 長さ | 内容 |
| --- | --- | --- | --- |
| データ区分 | N | 1 | "2" |
| 被仕向金融機関番号 | N | 4 |  |
| 被仕向金融機関名 | C | 15 | 半角カナ |
| 被仕向支店番号 | N | 3 |  |
| 被仕向支店名 | C | 15 | 半角カナ |
| 手形交換所番号 | C | 4 | 全スペース |
| 預金種目 | N | 1 |  |
| 口座番号 | N | 7 |  |
| 受取人名 | C | 30 | 半角カナ |
| 振込金額 | N | 10 | 右詰ゼロ埋め |
| 新規コード | N | 1 | "0" |
| 顧客コード1 | C | 10 | スペース |
| 顧客コード2 | C | 10 | スペース |
| 振込区分 | C | 1 | スペース |
| 識別表示 | C | 1 | スペース |
| ダミー | C | 7 | スペース |

### 6.4 トレーラレコード（区分=8）

| フィールド | 種別 | 長さ |
| --- | --- | --- |
| データ区分 | N | 1 |
| 合計件数 | N | 6 |
| 合計金額 | N | 12 |
| ダミー | C | 101 |

### 6.5 エンドレコード（区分=9）

| フィールド | 種別 | 長さ |
| --- | --- | --- |
| データ区分 | N | 1 |
| ダミー | C | 119 |

### 6.6 文字種ルール
- N (数値): ASCII数字、右詰ゼロ埋め
- C (文字): 半角カナ + 半角英大文字 + 半角数字 + スペース + 記号 `()-./, \`
- **半角カナフィールドに2バイト文字を入れない**（cp932で2バイトになる全角文字は除外）
- 長音・ダッシュ系8種（`ー`, `－`, `‐`, `‒`, `–`, `—`, `―`, `−`）は半角ハイフン `-` に統一

## 7. 主要処理フロー

### 7.1 PDF抽出フロー（`run_extraction` in `webapp/app.py`）

```
1. アップロードされたPDF群を受け取る
2. マスタから payee_hints（取引先名・カナのリスト）を構築
3. for each PDF:
     a. PyMuPDFでテキスト抽出（不足時はTesseract OCR）
     b. PDFをbase64エンコードしてClaude APIに送信
     c. tool_use レスポンスから InvoiceData を取得
     d. 例外時は空のInvoiceData (account_number=""、amount=0) で枠だけ作成
4. for each item:
     a. payee_db.find_match() でマスタ照合（口座番号→名義人カナ→取引先名）
     b. マッチした場合: invoice をマスタの値で上書き、matched_payee_key を記録
5. append=False: 新規バッチを作成して保存
   append=True: 既存バッチに追加（widget値を invoice に取り込んでから連結）
6. session_state["extracted_items"] に格納してrerun
```

### 7.2 全銀ファイル生成フロー（`run_generation` → `generate_zengin`）

```
1. session_state["current_consignor_id"] から振込元レコードを取得
2. 必須項目チェック（委託者コード・銀行・支店・口座番号）
3. 各itemのwidget値からInvoiceDataを再構築（手動編集を反映）
4. for each invoice:
     a. process_invoice() で銀行/支店コードを解決
     b. bank_resolver.get_bank_name_kana() で半角カナ名を取得（to_halfwidth_kana 経由）
5. merge_transfers() で同一振込先（銀行+支店+口座+種目）を合算
6. build_header / build_data × N / build_trailer / build_end を組み立て
7. cp932 でファイル出力
8. zengin_bytes を session_state に格納
9. auto_download_pending フラグを立てて rerun
10. rerun後、HTML注入でブラウザのダウンロードを自動開始
```

### 7.3 マスタ自動マッチングロジック（`payee_db.find_match`）

優先順位:
1. **口座番号完全一致** — 最も信頼度高い
2. **名義人カナ一致**（半角/全角を正規化して比較）
3. **取引先名一致**（空白を除いて部分一致）

最初にヒットした1件を返す（複数候補は出さない）。

### 7.4 バッチ自動保存タイミング
以下の操作で `save_current_batch_to_db()` が呼ばれる:
- PDF抽出完了直後（新規バッチ作成）
- 「現在のバッチにPDF追加」直後
- マスタ選択（`on_master_change`）
- 行削除
- 「全てクリア」
- 全銀ファイル生成
- 「現在の編集を保存」ボタン押下

手動編集（text_input等の入力中）は保存されない → 上記アクション or 明示保存が必要。

## 8. Claude API 連携

### 8.1 モデル
既定: `claude-sonnet-4-6`（設定可能）。

### 8.2 リクエスト構成
- `system`: 動的構築（ベースプロンプト + マスタヒント）
- `tools`: `extract_invoice_data` 1個のみ、`tool_choice` で強制呼び出し
- `messages.user.content`:
  - PDFをbase64で `document` ブロックとして送信
  - テキストで「この請求書から振込に必要な情報を抽出してください」

### 8.3 ツールスキーマ（`extract_invoice_data`）
出力フィールド: `payee_name` (required), `bank_name`, `bank_code`, `branch_name`, `branch_code`, `account_type` (enum), `account_number` (required), `amount` (required)

### 8.4 システムプロンプトの設計ポリシー
1. **「振込先（受取人）」と「請求書宛先（支払者）」を明確に区別** — 「○○様」「○○御中」は payee_name 不可
2. **括弧内の銀行/支店コードを必ず探す** — 「楽天銀行(0036)」のような表記
3. **金額誤抽出の防止** — 「3,615円×20%」のような値引き計算根拠は amount にしない
4. **レイアウト例とJSON正解を同梱** — Few-shot 学習効果
5. **マスタの取引先名一覧を動的追加** — 既知顧客のカナ表記が統一

### 8.5 フォールバック
API失敗時は `extract_with_regex` が動作:
- 金額: 「合計」「請求」「ご請求金額」等のキーワード周辺を正規表現で抽出
- 口座番号: 「普通口座 1234567」「口座番号: 1234567」等のパターン
- 銀行名・支店名・預金種目も類似のパターンマッチ

2. PDFアップロード + 振込元プルダウン + 振込種別 + 振込日(カレンダー) + 出力ファイル名 + APIキー
3. 「PDF読込・抽出（新規バッチ）」「現在のバッチに追加」「全てクリア」ボタン
4. 抽出済みアイテム数 + マスタ未登録件数のサマリー
5. 振込情報一覧（編集可、合計表示）
6. ワンクリック生成→ダウンロードボタン
7. 各PDFの「プレビュー画像 + 編集フォーム」（縦に並ぶ）
8. 全銀ファイル生成セクション（従来の二段階フロー）

### 9.3 各アイテム（PDFごと）の表示要素
- ヘッダー: `#N / ファイル名 / ページ数 / [✦マスタ一致 or ⊕マスタ未登録]`
- 「マスタに登録」ボタン（未登録のみ）/「行を削除」ボタン
- 「マスタから選択」プルダウン
- 左カラム: PDF1ページ目プレビュー画像（dpi=130）+ ページスライダー + PDFダウンロード
- 右カラム: 受取人名(text) / 銀行(select) / 銀行コード(自動) / 支店(select) / 支店コード(自動) / 種目(select) / 口座番号(text) / 金額(number)

### 10.1 秘匿情報の取り扱い

### 10.2 APIキー解決順序
1. `config.yaml` の `claude.api_key`
2. 環境変数 `ANTHROPIC_API_KEY`
3. `st.secrets["ANTHROPIC_API_KEY"]`

### 10.3 認証
- ローカル: 認証なし（`localhost` 限定アクセス）
- Streamlit Cloud: パスワードゲート（`require_password()`）。`APP_PASSWORD` Secret が未設定なら認証スキップ。

### 10.4 監査ポイント
- APIキーのローテーション周期（推奨3〜6ヶ月）
- Anthropic コンソールでの月次使用量上限設定（漏洩時の被害限定）
- Git履歴に機密が混入していないか定期チェック

## 12. 制約事項・既知の問題

### 12.1 Streamlit Community Cloud の制約
- データ永続化なし: コンテナ再起動で全SQLite DBがリセット → マスタは毎回CSVから再取り込みが必要
- 7日アクセスなしで sleep（次回起動に1〜2分）
- 1ワークスペースに private app は1つだけ（Public化 + パスワードゲートで運用）
- アップロード上限: 1ファイル200MB

### 12.2 Pythonバージョン互換
- f-string内のクォートネストは Python 3.12+ のみ（PEP 701）
- Streamlit Cloud は 3.11系のため、シングル/ダブルクォートをネストしない

### 12.3 Streamlit のモジュールリロード
- `webapp/app.py` の変更は自動検知してリロードされる
- `src/zengin_converter/` 配下や `webapp/*_db.py` などの依存モジュールはモジュールキャッシュが効き、自動リロードされない → サーバー再起動が必要

### 12.4 既知の制約
- 同一振込日で2バッチを作る場合、自動名前は連番（"YYYY年MM月DD日振込分 (2)"）
- フォーム手動編集は scriptが走るたびに保存しない（明示保存ボタン or 主要操作のタイミング）
- マスタの auto-match は最初の1件を返す（複数候補表示は未実装）
